# 圆桌协作（Roundtable Collaboration）

> **复杂任务：多个专家先讨论对齐，再分头实施。我是主持人。**

---

## 什么时候启动圆桌

以下条件满足时，必须加载圆桌技能：

1. 任务涉及 **2 个以上专业领域**（如前端+后端、设计+开发）
2. 专家之间有 **接口依赖**（一方输出是另一方的输入）
3. 需要 **多方对齐方案** 才能开始实施
4. 老板明确要求"让他们讨论一下"

## 前置分析（每次必做）

在启动圆桌之前，必须先完成以下分析：

### 1. 拆解任务所需技能
把任务按技术栈拆开：
- 需要什么语言/框架/工具？
- 需要什么专业能力？（设计/后端/数据库/安全/测试/写作...）
- 涉及几个层级？（前端/后端/数据库/部署/UI/内容...）

### 2. 检查现有员工池
看现有专家是否覆盖所有所需技能：
- 有 → 直接召集
- 没有 → 上网搜索该领域知识 → forge 铸造新专家 → 再召集

### 3. 搜索该领域最佳实践
在开始讨论前，先搜一下该领域的编程规范、架构模式、最佳实践。
让专家们在讨论时参考这些资料，不闷头造轮子。

### 4. 评估本地模型资源
本地模型慢（~10 t/s）或 GPU 不够 → 必须拆分任务为小块：
- 大模块拆成子任务
- 每完成一块验证一块
- 绝不允许一个大任务一把梭等结果

## 协作流程

### Round 0：我拆任务 & 选专家

```
老板：做个登录页面
  ↓
我：这个需要前端（页面）+ 后端（接口）+ 数据库（表）+ 设计师（交互）
  ↓
检查员工：都有 → 直接召集
  ↓
搜规范：搜一下登录功能的常见模式和最佳实践
  ↓
启动圆桌，邀请：前端工程师、后端工程师、数据库专家、设计师
```

### Round 1：各抒己见（并行）

我写一个统一 prompt，**同时**发给所有相关专家：

```python
# 用 execute_code 并行发 curl 给所有专家
import concurrent.futures

prompts = {
    '前端': '设计登录页面的前端结构...',
    '后端': '设计登录的 API 和数据表...',
    '设计师': '设计登录页面的交互流程...'
}

def call_expert(name, prompt):
    # 调本地模型
    return curl_llama(prompt)

with concurrent.futures.ThreadPoolExecutor() as ex:
    results = dict(ex.map(
        lambda kv: (kv[0], call_expert(*kv)),
        prompts.items()
    ))
```

**耗时（实测）：** ~160s（所有专家同时跑完，受最慢者约束）
**注意：** 每个专家需输出 500-800 tokens，~10 t/s 下需 50-80s。2 个专家并行不累加，就是最慢那个的时间。

### Round 2：交叉审查（串行）

我把各专家的方案互相给对方看：

```
前端 → 看到后端的 API 设计 → 说"这个接口字段不够"
后端 → 看到前端的方案 → 说"这个需求需要加个接口"
设计师 → 看到前后端方案 → 说"交互要改"
```

**实现方式：**
```
Step 1: 前端方案 → + 后端prompt → 发 curl → 后端反馈
Step 2: 后端方案 → + 前端prompt → 发 curl → 前端反馈
Step 3: 双方反馈 → + 设计师prompt → 发 curl → 设计师意见
```

**耗时：** ~90s（3 轮串行，每轮 ~30s）

### Round 3：汇总定案

把 Round 2 的反馈汇总，发给所有专家看最终方案。

### Round 4：分头实施

方案对齐后，各专家按自己的部分实施（走 expert 工作流）。

---

## 总耗时估算

| 环节 | 耗时（实测） | 说明 |
|------|------|------|
| Round 1：各抒己见 | ~160s | 并行，受最慢专家约束 |
| Round 2：交叉审查 | ~190s | 串行 3 轮 |
| Round 3：定案 | ~60s | 一次确认 |
| Round 4：实施 | ~150-200s | 每专家（并行） |
| **总计** | **~8-10 分钟** | **$0 成本** |

---

## 并行执行模板

推荐用 terminal(background=true) 分别启动每个请求，再逐个收集结果。不要用 execute_code 做并行（300s 硬超时限制）。

每步调用间保存中间结果到 /tmp/round*.txt，方便后续轮次读取。

---

## 冲突处理

| 冲突类型 | 处理方式 |
|----------|----------|
| 方案分歧 | 我(主持人)裁决，选择技术更优方案 |
| 接口不对齐 | 让双方各让一步，取简单方案 |
| 超时未响应 | 默认同意对方方案，继续推进 |
| 连续 2 轮无法对齐 | 升级给主人决策 |

---

## 限制与注意事项

1. **总耗时约 8-10 分钟**：比单专家慢 3-5 倍，但免费。
2. **不适合紧急任务**：急用的任务别走圆桌，直接指派一个专家。
3. **专家越多越慢**：建议一次最多 3 个专家参与讨论。
4. **每次讨论后更新 learnings**：记录协作教训，下次更快。
5. **Shell 引号是最大坑**：prompt 里含中文时，一定用 -d @/tmp/payload.json。
6. **reasoning_content 陷阱**：Gemma 4 E4B 在复杂中文 prompt 下会吐到 reasoning。用英文 system prompt 可缓解。
